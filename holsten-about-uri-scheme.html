Abstract

This document specifies the URI (Uniform Resource Identifier) scheme "about". About URIs are designed to be an internal, application-level identifier. Unlike many other URI schemes, the resolution of, and resources represented by, about URIs are left entirely to each individual application.


1.  Introduction

An about URI is designed to be used internally by applications for almost any desired purpose. Such URIs have commonly been used by web browsers for proving access to built-in functionality, such as application preferences and settings, information about the application, or "easter eggs".

About URIs are more memorable than an equivalent urn or tag URI. Using an http URI would not only be less memorable, but also longer and semantically confusing. While any number of existing schemes could be used to identify such resources, about URIs have become the de facto standard. Browsers already use the about scheme. The about:blank (about:blank) URI is ubiquitous, and some browsers also provide other resources with the about scheme, including those in Section 3.2 (Examples).[NOTE-1] (JH: It may be reasonable to also define an about URN scheme and define about URIs as a provisional shorthand for those URNs.)

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC2119] (Bradner, S., “Key words for use in RFCs to Indicate Requirement Levels,” March 1997.).


2.  URI Syntax

The general syntax of a about URI, in ABNF (Crocker, D. and P. Overell, “Augmented BNF for Syntax Specifications: ABNF,” January 2008.) [RFC5234] is: 

          abouturi = "about:" segment

where "segment" is imported from [RFC3986] (Berners-Lee, T., Fielding, R., and L. Masinter, “Uniform Resource Identifier (URI): Generic Syntax,” January 2005.). About URIs are always escaped, as per [RFC3986] (Berners-Lee, T., Fielding, R., and L. Masinter, “Uniform Resource Identifier (URI): Generic Syntax,” January 2005.).

No relative URI syntax is defined.


3.  Resolving About URIs

In general, applications are free to resolve any about URI to any resource, either internal or external, or redirect to an alternative URI, with about:blank (about:blank) being the only exception.

As about URIs are designed to be internal to each application, there is no expectation of any URI, except about:blank (about:blank), returning the same resource among different applications. However, it is worth noting that some conventions have arisen for providing particular functionality via common about URIs.

Because about URIs identify application specific information, applications should not need to retrieve remote information for such a resource.[NOTE-2] (JH: draft when an application should create a new about URI, how to handle unknown about URIs)

The about:blank (about:blank) URI is the only about URI reserved by this specification.[NOTE-3] (JH: Should undefined about URIs redirect to about:blank?)


3.1.  about:blank

This URI identifies an empty resource. If the application may use a document of MIME type 'text/html' and character encoding 'UTF-8', about:blank SHOULD be represented with an empty document. Other representations are not defined.


3.2.  Examples

The following examples illustrate some known URIs supported by existing applications. They are not guaranteed to be resolvable by every application.

about:config
Commonly provides access to application preferences and settings
about:cache
Commonly provides access to information about resources stored in the browsers cache.
about:plugins
Commonly provides access to information about installed plugins
about:mozilla
An easter egg supported by Mozilla showing a passage from the fictional Book of Mozilla
about:internets
An easter egg supported by Google Chrome depicting the internet as a series of tubes.

5.  Security Considerations

There is no guarantee that an application will understand any about URI provided to it. An about URI may not resolve to the expected resource. If the reference is unlikely to resolve correctly, the reference should be accompanied by an explanation or alternatives.

Some applications are known to display arbitrary HTML following the scheme. Applications should be cautious when accessing an about URI. An application should not execute or display information in an about URI. About URIs should not cause the application to modify any data. Applications should not use about URIs to access files or other sensitive information.

About URIs may identify resources which show private information. This sensitive information should never be included in about URIs.[NOTE-5] (JH: convert to use meanings from RFC2119.)[NOTE-6] (LH: insert information about Origin)
